## 3. openspec_plan sub-workflow

- [x] 3.1 Add `coding-harness/workers/workflows/openspec_plan.json`: scaffold via `openspec_new_change`, then a bounded outer `DO_WHILE` loop (capped by `openspecMaxIterations`), each pass draining the artifact dependency graph via a **`SUB_WORKFLOW` call to `openspec_artifact_drain.json`** (`openspec_status` → JQ-select not-yet-generated-this-pass artifacts with `status != "blocked"` → `FORK_JOIN_DYNAMIC` over `openspec_generate_artifact` sub-workflows → merge/accumulate progress, looping internally until nothing new is selectable). Added `openspec_generate_artifact.json` as the per-artifact fork unit (`openspec_instructions` → build prompt → `coding_agent`).
- [x] 3.2 Port `design_docs.json`'s `review_mode` SWITCH verbatim into the loop: `openspecHumanApproval` true → `HUMAN` task capturing `approved`/`feedback` (like today's `design_review`); default → read-only AI-judge `coding_agent` task returning `{approved, feedback}` via structured output (like today's `design_judge`, extended to also judge tasks.md's independence/file-disjointness). Rejection (from either branch) loops back to regenerate every artifact (the drain sub-workflow naturally regenerates everything once nothing is `blocked`); loop continues while not approved and the iteration cap isn't exhausted.
- [x] 3.3 Wire loop exit: approval commits the change dir and routes to `openspec_tasks_to_subtasks`, setting `openspec_plan`'s `outputParameters.subtasks` to its result; exhausting `openspecMaxIterations` without approval routes to `TERMINATE` with the last feedback in `workflowOutput` (matching `design_docs.json`'s `design_not_approved` case). Registered all three new sub-workflows in `register.sh`'s ordered list (`openspec_generate_artifact`, `openspec_artifact_drain`, then `openspec_plan`, before `code_parallel`), replacing the removed `design_docs` entry.
- [x] 3.4 **Found and fixed via live testing**: Conductor does not support a `DO_WHILE` nested directly inside another `DO_WHILE`'s `loopOver` — the original 3.1 inlined the drain loop as a nested `DO_WHILE`, which silently stopped advancing after the first parallel fan-out (no exception, on both SQLite- and Postgres-backed servers, ruling out a database cause). Extracted the drain loop into its own `openspec_artifact_drain.json` sub-workflow (per the product docs: "use a Sub Workflow task inside the Do While task" instead of nesting). Added a general regression test (`test_no_do_while_nested_directly_inside_do_while`) walking every workflow JSON to catch this class of bug in the future. Re-verified live end-to-end after the fix — see 6.2/6.3.
